# Niryo NED2 and ROS2

The Niryo NED2 runs a ROS1-based control stack onboard. To control it from a our ROS2 development machine, Niryo provides an official ROS2 driver: [`ned-ros2-driver`](https://github.com/NiryoRobotics/ned-ros2-driver). This is maintained by Niryo themselves and is the supported path for ROS2 integration.

## Architecture

The driver runs on your development machine and connects to the ROSBridge WebSocket server already running on the robot:

```
ROS2 Client (development machine)
    ↔ ned-ros2-driver (ROS2 node, development machine)
    ↔ ROSBridge WebSocket Server — port 9090 (on NED2)
    ↔ Niryo ROS1 Stack — ned_ros (robot firmware)
```

The driver bridges topics, services, and actions transparently: names and namespaces are preserved exactly as they exist on the ROS1 side. Only subscribed interfaces are bridged, keeping network overhead low.

## Installation

we will use our dedicated workspace from earlier to avoid conflicts with other packages:

```bash
mkdir -p ~/mars_ned2/src
cd ~/mars_ned2/src
git clone https://github.com/NiryoRobotics/ned-ros2-driver.git

cd ~/mars_ned2
rosdep install --from-paths src --ignore-src -r -y
```

A virtual environment with system site-packages access keeps Python dependencies isolated:

```bash
python3 -m venv venv --system-site-packages
source venv/bin/activate
pip install -r src/ned-ros2-driver/requirements.txt
```

Build:

```bash
source /opt/ros/jazzy/setup.bash
colcon build
source install/setup.bash
```

The driver has been tested with ROS2 Jazzy and Python 3.12.

## Network Configuration

Edit `config/drivers_list.yaml` to point at your robot. 

**Single robot, no namespace:**

```yaml
rosbridge_port: 9090
robot_namespaces:
  - ""
robot_ips:
  - "192.168.1.100"
```

**Multiple robots with namespaces:**

```yaml
rosbridge_port: 9090
robot_namespaces:
  - "arm1"
  - "arm2"
robot_ips:
  - "192.168.1.101"
  - "192.168.1.102"
```

With namespaces, all interfaces are prefixed: `/arm_1/joint_states`, `/arm_2/joint_states`, etc.

## Launching the Driver

```bash
source ~/mars_ned2/install/setup.bash
ros2 launch niryo_ned_ros2_driver driver.launch.py
```

Verify the bridge is up:

```bash
ros2 node list    
ros2 topic list
ros2 service list
ros2 action list
```


## Driver Capabilities

All commands below talk to the robot's ROS1 stack ([`ned_ros`](https://github.com/NiryoRobotics/ned_ros)) through the driver. The interface documentation lives at [niryorobotics.github.io/ned_ros](https://niryorobotics.github.io/ned_ros/).

### Calibration

The stepper motors on joints 1–3 require calibration on startup. This is a Ned2-only service. `value: 1` triggers automatic calibration.

```bash
ros2 service call /niryo_robot/joints_interface/calibrate_motors \
  niryo_ned_ros2_interfaces/srv/SetInt \
  "{value: 1}"
```

### TCP (Tool Center Point)

The TCP defines the transform from the `tool_link` frame to the effective tool tip. When enabled, all Cartesian moves are performed relative to this point, so the same program works across tools by swapping the TCP.

**Set TCP** — defines position (metres) and orientation (quaternion) of the TCP relative to `tool_link`:

```bash
ros2 service call /niryo_robot_tools_commander/set_tcp \
  niryo_ned_ros2_interfaces/srv/SetTCP \
  '{position: {x: 0.0726, y: -0.0007261, z: -0.01155}, orientation: {x: 0.0, y: 0.0, z: 0.0, w: 1.0}, tcp_version: 2}'
```

**Enable TCP** — activates (or deactivates) the TCP. When disabled, the TCP coincides with `tool_link`:

```bash
ros2 service call /niryo_robot_tools_commander/enable_tcp \
  niryo_ned_ros2_interfaces/srv/SetBool \
  '{value: true}'
```

**Reset TCP** — restores the default TCP for the currently equipped tool:

```bash
ros2 service call /niryo_robot_tools_commander/reset_tcp \
  niryo_ned_ros2_interfaces/srv/Trigger "{}"
```

### TF Utilities

**Publish a static transform** (e.g. place a target frame 0.3 m in front of and 0.2 m above the world origin):

```bash
ros2 run tf2_ros static_transform_publisher 0.3 0.0 0.2 0 0 0 world target
```

**Camera link** — approximate static transform from `wrist_link` to the camera frame (yaw ≈ 100°):

```bash
ros2 run tf2_ros static_transform_publisher 0.0385 0 0 0 0 1.7453 wrist_link camera
```


### Robot State

Read the current end-effector pose, tool status, and more:

```bash
ros2 topic echo /niryo_robot/robot_state --once
```

Topic type: `niryo_ned_ros2_interfaces/msg/RobotState`

### Motion Control

All motion commands go through the `/niryo_robot_arm_commander/robot_action` action server using `niryo_ned_ros2_interfaces/action/RobotMove`. The `cmd_type` field selects the motion mode:

| `cmd_type` | Mode |
|---|---|
| `0` | Move joints |
| `1` | Move to pose (joint-space plan) |
| `5` | Move to pose linear (Cartesian straight line) |

**Move joints** — specify all 6 joint angles in radians:

```bash
ros2 action send_goal /niryo_robot_arm_commander/robot_action \
  niryo_ned_ros2_interfaces/action/RobotMove \
  "{cmd: {cmd_type: 0, joints: [0.0, 0.57, 1.57, 0.0, 0.0, 0.0]}}"
```

**Move to pose** — joint-space planned move to a Cartesian pose (position in metres, RPY in radians):

```bash
ros2 action send_goal /niryo_robot_arm_commander/robot_action \
  niryo_ned_ros2_interfaces/action/RobotMove \
  "{cmd: {cmd_type: 1, position: {x: 0.3, y: 0.0, z: 0.2}, rpy: {roll: 0.0, pitch: 0.0, yaw: 0.0}}}"
```

**Move to pose linear** — Cartesian straight-line move to a pose. Uses the same fields as above but follows a linear path in task space:

```bash
ros2 action send_goal /niryo_robot_arm_commander/robot_action \
  niryo_ned_ros2_interfaces/action/RobotMove \
  "{cmd: {cmd_type: 5, position: {x: 0.3, y: 0.0, z: 0.2}, rpy: {roll: 0.0, pitch: 0.0, yaw: 0.0}}}"
```

**Move XYZ only** — use `cmd_type: 1` (or `5` for linear) and set RPY to zeros. The robot will reach the target position with a neutral wrist orientation:

```bash
ros2 action send_goal /niryo_robot_arm_commander/robot_action \
  niryo_ned_ros2_interfaces/action/RobotMove \
  "{cmd: {cmd_type: 1, position: {x: 0.3, y: 0.0, z: 0.25}, rpy: {roll: 0.0, pitch: 0.0, yaw: 0.0}}}"
```

If you need to reach a position regardless of orientation, consider computing IK first (see below) and issuing a joint move instead.

### Kinematics

**Forward kinematics** — joint angles → end-effector pose:

```bash
ros2 service call /niryo_robot/kinematics/forward \
  niryo_ned_ros2_interfaces/srv/GetFK \
  "{joints: [0.0, 0.57, 1.57, 0.0, 0.0, 0.0]}"
```

**Inverse kinematics** — Cartesian pose → joint angles:

```bash
ros2 service call /niryo_robot/kinematics/inverse \
  niryo_ned_ros2_interfaces/srv/GetIK \
  "{pose: {position: {x: 0.3, y: 0.0, z: 0.3}, rpy: {roll: 0.0, pitch: 1.57, yaw: 0.0}}}"
```

### Tool Control

Tool commands go through `/niryo_robot_tools_commander/action_server` using `niryo_ned_ros2_interfaces/action/Tool`. The `cmd_type` field selects the operation and `tool_id` identifies the attached tool.

**Tool IDs:**

| Tool | ID |
|---|---|
| Standard gripper | 11 |
| Large gripper | 12 |
| Adaptive gripper | 13 |
| Electromagnet | 30 |
| Vacuum pump | 31 |
| Vacuum pump v2 | 32 |

**Open gripper** (`cmd_type: 1`):

```bash
ros2 action send_goal /niryo_robot_tools_commander/action_server \
  niryo_ned_ros2_interfaces/action/Tool \
  "{cmd: {cmd_type: 1, tool_id: 11, max_torque_percentage: 80, hold_torque_percentage: 30}}"
```

**Close gripper** (`cmd_type: 2`):

```bash
ros2 action send_goal /niryo_robot_tools_commander/action_server \
  niryo_ned_ros2_interfaces/action/Tool \
  "{cmd: {cmd_type: 2, tool_id: 11, max_torque_percentage: 80, hold_torque_percentage: 30}}"
```

`max_torque_percentage` controls closing force; `hold_torque_percentage` controls the torque used to hold the grip once closed.

## Key Interface Reference

### Topics

| Topic | Type | Description |
|---|---|---|
| `/joint_states` | `sensor_msgs/msg/JointState` | Joint positions, velocities, efforts |
| `/tf` / `/tf_static` | `tf2_msgs/msg/TFMessage` | Transform tree |
| `/niryo_robot_tools_commander/current_id` | `std_msgs/msg/Int32` | Currently detected tool ID |
| `/niryo_robot_tools_commander/tcp` | `niryo_ned_ros2_interfaces/msg/TCP` | TCP state and transform |

### Services

| Service | Type | Description |
|---|---|---|
| `/niryo_robot/joints_interface/calibrate_motors` | `niryo_ned_ros2_interfaces/srv/SetInt` | Calibrate stepper motors (Ned2 only; `value: 1` = auto) |
| `/niryo_robot_tools_commander/set_tcp` | `niryo_ned_ros2_interfaces/srv/SetTCP` | Set and activate a new TCP transform |
| `/niryo_robot_tools_commander/enable_tcp` | `niryo_ned_ros2_interfaces/srv/SetBool` | Enable or disable TCP |
| `/niryo_robot_tools_commander/reset_tcp` | `niryo_ned_ros2_interfaces/srv/Trigger` | Reset TCP to tool default |
| `/niryo_robot/kinematics/forward` | `niryo_ned_ros2_interfaces/srv/GetFK` | Forward kinematics |
| `/niryo_robot/kinematics/inverse` | `niryo_ned_ros2_interfaces/srv/GetIK` | Inverse kinematics |


### Actions

| Action | Type | Description |
|---|---|---|
| `/niryo_robot_arm_commander/robot_action` | `niryo_ned_ros2_interfaces/action/RobotMove` | High-level arm motion (joints, pose, linear) |
| `/niryo_robot_tools_commander/action_server` | `niryo_ned_ros2_interfaces/action/Tool` | Tool command execution |
| `/niryo_robot_follow_joint_trajectory_controller/follow_joint_trajectory` | `control_msgs/action/FollowJointTrajectory` | Low-level joint trajectory execution |


## Troubleshooting 

here are some issued that occured occasionally

**Cannot connect to ROSBridge:**
The bridge runs on the robot at port 9090. If it is not reachable, Niryo Studio uses the same connection, try using that to connect first.  If Studio connects, the port is open. Otherwise, restart the robot's ROS stack.

**MoveIt2 timing errors:**
MoveIt2 rejects trajectories whose timestamps fall outside the valid window. This usually means the robot's clock timezone is wrong. Fix:

```bash
ssh niryo@<robot_ip>
timedatectl set-timezone Europe/London
sudo systemctl restart systemd-timesyncd
```

Then restart the robot software.
