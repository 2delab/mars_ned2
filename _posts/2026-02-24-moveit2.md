# Getting MoveIt2 Working: From Setup Assistant to Collision-Aware Motion Planning

## What MoveIt2 Actually Is

MoveIt2 is a stack of middleware components sitting between high-level task commands and low-level robot joint controllers. 
When code says "move end effector to position X,Y,Z," MoveIt2 handles the execution and engineering:
- **Inverse kinematics (IK):** Converts Cartesian poses into joint angles
- **Motion planning:** Finds collision-free paths through joint space from current state to goal
- **Collision checking:** Tests every trajectory point against the robot itself, obstacles, and the environment
- **Planning scene management:** Maintains a live 3D model of the workspace

## The Setup Assistant: A Map Through the Maze
 The Setup Assistant provides a GUI wizard generating the MoveIt2 configuration package structure automatically.


#### Launching the Setup Assistant

```bash
ros2 launch moveit_setup_assistant setup_assistant.launch.py
```

Select **Create New MoveIt Configuration Package**, browse to your robot's URDF file, and click **Load Files**.

## Configuration Steps
### 1. Generate Self-Collision Matrix
Navigate to **Self-Collisions** pane. Adjust sampling density to maximum for accuracy, then click **Generate Collision Matrix**. The tool checks 10,000+ random positions to identify link pairs that can safely skip collision checking. Review and manually adjust the generated matrix as needed.

![move_it_setup_assistant](/mars_ned2/assets/images/mv3.png){: .align-center}

### 2. Configure Planning Groups
Planning groups define semantic robot components. Create groups for each controllable subsystem:

**For serial chain manipulators:**
- Group name: `arm_name`
- Kinematic solver: `kdl_kinematics_plugin/KDLKinematicsPlugin`
- Click **Add Joints** → select joint chain → **Save**

**For end effectors:**
- Group name: `gripper_name`  
- Kinematic solver: **None** (non-serial links)
- Click **Add Links** → select gripper links → **Save**

![move_it_setup_assistant](/mars_ned2/assets/images/mv4.png){: .align-center}
![move_it_setup_assistant](/mars_ned2/assets/images/mv5.png){: .align-center}


### 3. Configure ros2_control
If your URDF lacks ros2_control tags:
- Navigate to **ros2_control URDF Modification**
- Select command interfaces (default: `position`)
- Select state interfaces (default: `position`)
- Click **Add Interfaces**

![move_it_setup_assistant](/mars_ned2/assets/images/mv6.png){: .align-center}

### 4. Setup ROS 2 Controllers
Auto-generate simulated controllers:

**Joint trajectory controller:**
- Controller name: `arm_controller`
- Type: `joint_trajectory_controller/JointTrajectoryController`
- Add joints via **Add Planning Group Joints**

**Gripper controller:**
- Controller name: `gripper_controller`
- Type: `position_controllers/GripperActionController`
- Add gripper group joints

![move_it_setup_assistant](/mars_ned2/assets/images/mv7.png){: .align-center}

### 5. Configure MoveIt Controllers
Create MoveIt-specific controller interfaces:

**For arms:**
- Controller name: matches ROS 2 controller
- Type: `FollowJointTrajectory`
- Select arm planning group

**For grippers:**
- Controller name: matches ROS 2 controller  
- Type: `Gripper Command`
- Select gripper planning group

![move_it_setup_assistant](/mars_ned2/assets/images/mv8.png){: .align-center}


### 6. Generate Package
Final steps:
1. **Author Information**: Enter name and email
2. **Configuration Files**: Browse to workspace `src` directory
3. Create folder for package (e.g., `robot_moveit_config`)
4. Click **Generate Package**

![move_it_setup_assistant](/mars_ned2/assets/images/mv9.png){: .align-center}

## Key Considerations
- **Sampling density**: Higher values improve collision matrix accuracy but increase computation time
- **Joint limits**: Test all joints in Robot Poses step to verify URDF limits
- **Controller naming**: Maintain consistency between ROS 2 and MoveIt controller names
- **Kinematics solvers**: Consider IKFast or pick_ik for improved performance over default KDL

The generated package includes launch files, configuration YAML files, and the SRDF—everything needed to integrate your robot with MoveIt's motion planning framework.

## Inside the Generated Config: What Actually Matters
The Setup Assistant generates files. Editing them becomes necessary. Here's what each does:
### `.srdf` (Semantic Robot Description Format)
Robot semantics—planning groups, collision rules, and joint limits as metadata layered on the URDF.
Example excerpt from generated `ned2.srdf`:
```xml
<robot name="ned2">
  <group name="arm">
    <joint name="joint_1"/>
    <joint name="joint_2"/>
    <!-- ... joint_3 through joint_6 ... -->
  </group>
  <disable_collisions link1="shoulder" link2="upper_arm" reason="adjacent"/>
  <disable_collisions link1="upper_arm" link2="forearm" reason="adjacent"/>
  <!-- ... more collision pairs ... -->
</robot>
```
Constant editing during development. This file encodes the semantic understanding of robot structure.
### `kinematics.yaml`
Specifies IK solver for each planning group:
```yaml
arm:
  kinematics_solver: kdl_kinematics_plugin/KDLKinematicsPlugin
  kinematics_solver_search_resolution: 0.005
  kinematics_solver_timeout: 0.005
  kinematics_solver_attempts: 3
```
KDL (Kinematics and Dynamics Library) ships with MoveIt2 and works generically for serial manipulators. Not fast—5 milliseconds per IK query—but reliable.
Production systems with non-standard arm geometry would implement custom IK plugins. The generic solver sufficed for sorting.
### `joint_limits.yaml`
Hardware constraints:
```yaml
joint_limits:
  joint_1:
    has_velocity_limits: true
    max_velocity: 2.618  # rad/s, ~150°/s
    has_acceleration_limits: true
    max_acceleration: 5.0
```
These come from manufacturer spec sheets. Niryo's documentation provided limits. MoveIt2 uses these to _time_ motions—respecting hardware capabilities during planning.
### `controllers.yaml`
Maps planning group names to actual ROS2 controllers:
```yaml
moveit_controller_manager: moveit_simple_controller_manager/SimpleControllerManager
moveit_simple_controller_manager:
  controller_names:
    - joint_state_controller
    - ned2_arm_controller
    - ned2_gripper_controller
  ned2_arm_controller:
    action_ns: /ned2_arm_controller/follow_joint_trajectory
    type: FollowJointTrajectory
    default: true
    joints:
      - joint_1
      - joint_2
      - joint_3
      - joint_4
      - joint_5
      - joint_6
```
This provides the _wiring_. MoveIt2 generates joint trajectories; the controller manager sends them to actual hardware. The action namespace must match what Niryo drivers expose.

## Launching MoveIt2 and What Actually Happens
The Setup Assistant generates `move_group.launch.py`:
```python
from moveit_configs_utils import MoveItConfigsBuilder
from moveit_configs_utils.launches import generate_move_group_launch
def generate_launch_description():
    moveit_config = MoveItConfigsBuilder(
        robot_name="ned2",
        package_name="ned2_moveit_config"
    ).to_moveit_configs()
    return generate_move_group_launch(moveit_config)
```
Launching it:
```bash
ros2 launch ned2_moveit_config move_group.launch.py
```
Several things occur in sequence:
1. **Joint State Broadcaster starts.** Listens to `/joint_states`, populates MoveIt2's internal model of current robot configuration.
2. **Planning Scene Monitor starts.** Creates 3D collision model from URDF/SRDF, listens to `/tf` for dynamic obstacles.
3. **Move Group Node starts.** Core service provider advertising:
   - `/move_group/plan_kinematic_path` (planning service)
   - `/move_group/execute_trajectory` (execution service)
   - `/move_group/get_planning_scene` (query scene state)
   - and more.
Verification:
```bash
ros2 service list | grep move_group
```
Response:
```
/move_group/apply_planning_scene
/move_group/clear_octomap
/move_group/execute_trajectory
/move_group/get_cartesian_path
/move_group/get_planning_scene
/move_group/plan_kinematic_path
/move_group/query_trajectories
/move_group/set_planning_scene_diff
```

## The Real Architecture: Where MoveIt2 Fits
System architecture understanding:
```
Sorting Application (task logic)
           ↓
    Motion Planning Client (application code)
           ↓
    /move_group ROS2 Services
           ↓
    [MoveIt2 Stack]
    ├─ OMPL Planner (RRT, PRM, etc.)
    ├─ Collision Checker (FCL library)
    ├─ IK Solver (KDL)
    └─ Planning Scene (3D model + transforms)
           ↓
    Controller Manager
           ↓
    Robot Driver (Niryo ROS2 stack)
           ↓
    Physical Hardware
```
MoveIt2's responsibility: taking "move end effector here" and returning a collision-free joint trajectory.
The planning scene is the key innovation—a live 3D model of the workspace:
- Loads robot geometry from URDF
- Loads planning constraints from SRDF
- Watches `/tf` for dynamic object updates
- Queries collision library (FCL - Flexible Collision Library) for every trajectory point
This architecture explains why joint states and TF publishing are critical. Without them, MoveIt2 operates with stale or missing world state.

## Performance Considerations
### Planning Time
With default RRTConnect planner on Niryo Ned2 (6 DOF):
- Simple motions in free space: 50-200ms
- Motions near obstacles: 200-1000ms
- Complex constrained motions: 1-5 seconds

### Collision Checking Overhead
For each trajectory point, FCL checks:
- Self-collision (robot against itself)
- Environment collision (robot against obstacles)
With detailed mesh geometry:
- ~50 microseconds per collision check
- Typical trajectory: 100-200 points
- Total overhead: 5-10ms per trajectory
Simplified collision geometry (primitive shapes instead of meshes) reduces this by 10x.
nned paths in 3D revealed issues invisible in terminal output.

### Planning Scene Monitoring
Real-time planning scene visualization:
```bash
ros2 topic echo /move_group/monitored_planning_scene
```
Shows all collision objects, attached objects, and allowed collision matrices. Critical for verifying obstacle detection.
