## Physics simulation

Gazebo is a robotics simulator that runs a full virtual environment. It can simulate physics, sensors and time while integrating natively with ROS 2. Unlike a pure dynamics engine, Gazebo can publish on the same topics, services, and TF tree that a real robot would: joint states come out of `/joint_states`, the clock drives `use_sim_time`, and the controller manager talks to a hardware plugin that writes into the physics engine. From the perspective of the rest of the software stack, simulation and hardware are indistinguishable.

## Use case

The physical Ned2 robots aren't always accessible, and running an untested control pipeline directly on hardware is a bad idea. Before sending trajectories to real motors, you need to know that the URDF is correct, the `ros2_control` stack loads cleanly, the controller manager finds its hardware backend, and MoveIt can plan and execute without the whole system falling over on startup.

Gazebo lets you validate all of that without touching the robot. Launch ordering bugs, missing inertial tags, clock synchronization issues, and controller sequencing problems are all much cheaper to discover in simulation. Once the pipeline is solid in sim, the same launch files and the same controller configuration transfer directly to the physical robot — which is the goal.

# From URDF to Gazebo Simulation


Getting the Niryo Ned2 running in Gazebo with full ros2_control means more than just launching a simulator. It required coordinating environment configuration, robot description generation, the simulator itself, a clock bridge, a robot spawner, and a controller stack, exactly  in the right order.

## Environment and Robot Description

Before anything starts, two environment variables are set. `GZ_SIM_RESOURCE_PATH` points Gazebo at the parent directory of `niryo_robot_description` so it can resolve the relative mesh paths baked into the URDF. The robot spawns broken without it. `RCL_LOGGING_CONFIG_DIR` is set to an empty string to suppress the wall of RCL output that otherwise floods the terminal on startup.

The robot description is generated at launch time rather than pre-baked, using `xacro` via ROS 2's `Command` substitution:

```python
robot_description_content = Command(
    [
        PathJoinSubstitution([FindExecutable(name="xacro")]),
        " ",
        PathJoinSubstitution(
            [niryo_description_share, "urdf", "ned2", "niryo_ned2_gazebo.urdf.xacro"]
        ),
        " controllers_config_file:=",
        PathJoinSubstitution(
            [niryo_controllers_share, "config", "ros2_controllers.yaml"]
        ),
    ]
)
```

`Command([...])` executes lazily when the launch file runs. The result is wrapped in `ParameterValue(..., value_type=str)` and passed to `robot_state_publisher`, which publishes it on `/robot_description` — the single source of truth for both the spawner and MoveIt. The `controllers_config_file` argument is critical: it's passed into the xacro so the resulting URDF embeds the correct `<ros2_control>` hardware configuration, including controller types, joint names, and interface definitions.

## Starting Gazebo

Gazebo is launched by including `ros_gz_sim`'s own `gz_sim.launch.py` rather than calling `gz sim` directly, which ensures the full internal lifecycle of the simulator is respected:

```python
IncludeLaunchDescription(
    PythonLaunchDescriptionSource(
        PathJoinSubstitution(
            [FindPackageShare("ros_gz_sim"), "launch", "gz_sim.launch.py"]
        )
    ),
    launch_arguments=[
        ("gz_args", [" -r -v ", gazebo_verbosity, " ", gazebo_world]),
        ("use_sim_time", use_sim_time),
    ],
),
```

`-r` starts the simulation running immediately rather than opening paused. `-v` sets verbosity level 4 during development gives the most output. The world defaults to Gazebo's built-in `empty.sdf` but is overridable from the command line via `gazebo_world`.

## Clock Bridge

The clock bridge starts immediately in parallel with Gazebo because any node that tries to register with the controller manager before a clock source exists will hang:

```python
clock_bridge = Node(
    package="ros_gz_bridge",
    executable="parameter_bridge",
    arguments=["/clock@rosgraph_msgs/msg/Clock[gz.msgs.Clock"],
    output="screen",
    parameters=[{"use_sim_time": use_sim_time}],
)
```
 `use_sim_time: true` is passed to every node in the launch file so all clocks are synchronized to Gazebo's simulation time rather than wall time.

## Spawning the Robot

The robot is spawned into Gazebo using `ros_gz_sim`'s `create` executable, reading from the `/robot_description` topic rather than a file:

```python
spawn_entity = Node(
    package="ros_gz_sim",
    executable="create",
    arguments=[
        "-topic", "robot_description",
        "-name", "niryo_ned2",
        "-allow_renaming", "true",
    ],
)

delayed_spawn_entity = TimerAction(period=5.0, actions=[spawn_entity])
```

Using `-topic` guarantees the spawner uses the exact same description `robot_state_publisher` already has. The five-second `TimerAction` delay is deliberate — Gazebo needs time to fully initialize its physics engine and world before it can accept spawn requests. Reducing it risks the spawn request arriving before the service is ready.

## Controller Sequencing

Controllers come up last, chained via `RegisterEventHandler` with `OnProcessExit` so each step only begins after the previous one fully completes:

```python
RegisterEventHandler(
    event_handler=OnProcessExit(
        target_action=spawn_entity,
        on_exit=[TimerAction(period=1.0, actions=[joint_state_broadcaster])],
    )
),
RegisterEventHandler(
    event_handler=OnProcessExit(
        target_action=joint_state_broadcaster,
        on_exit=[TimerAction(period=1.0, actions=[joint_trajectory_controller])],
    )
),
```

`joint_state_broadcaster` must be active before `niryo_ned2_arm_controller` is spawned as the trajectory controller needs joint state feedback to initialize. `OnProcessExit` fires when the `spawner` executable exits after successfully registering the controller with `/controller_manager`, making it a reliable sequencing signal compared to a fixed timer.

## The Missing Piece

With all of that running, the robot spawns and the controllers register but sending a joint trajectory does nothing. The controller manager has no hardware backend to write into. Two additions are required so we create a copy of the original urdf file  `niryo_ned2_gazebo.urdf.xacro` to add:

```xml
<!-- inside the <ros2_control> block -->
<ros2_control name="GazeboSimSystem" type="system">
  <hardware>
    <plugin>gz_ros2_control/GazeboSimSystem</plugin>
  </hardware>
  <!-- joint command and state interfaces -->
</ros2_control>

<!-- at the top level, alongside other <gazebo> tags -->
<gazebo>
  <plugin filename="libgz_ros2_control-system.so"
          name="gz_ros2_control::GazeboSimROS2ControlPlugin">
    <parameters>/path/to/ros2_controllers.yaml</parameters>
  </plugin>
</gazebo>
```

`GazeboSimSystem` is the hardware plugin that gives the controller manager a backend. commands are written into a void without it. `GazeboSimROS2ControlPlugin` is what instantiates the controller manager inside Gazebo's plugin context in the first place. Without both, the control pipeline is severed at the physics layer.

## Why not just use RViz

A physics engine adds no practical value for this project. we can use `mock_components/GenericSystem`, part of `ros2_control`'s standard library, instead of the Gazebo hardware plugin giving us the full controller stack and MoveIt a working interface with no simulator overhead. Everything validated in RViz transfers directly to the real robot. additionally Gazebo require modifying the urdf,  and pulling in `gz_ros2_control` as a dependency which adds unecessary complexitty.

## Common Pitfalls

**VM Rendering: Switching from OGRE 2 to OGRE 1**

Running Gazebo inside a virtual machine often produces flashing and corrupted visuals. The default render engine in modern Gazebo (Harmonic) is OGRE 2, which requires OpenGL 3.3+ features that most VM graphics drivers do not properly emulate.

After saving and restarting Gazebo, rendering is stable inside the VM.

**Forgetting `use_sim_time`** on any node that processes sensor data or publishes transforms causes timestamp mismatches that break tf2, navigation, and MoveIt.

**Missing inertial tags** in the URDF will cause Gazebo to silently misbehave — links without `<inertial>` blocks may spawn but won't respond correctly to forces or joint commands.
