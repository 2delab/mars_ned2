## Introduction to PyNiryo

PyNiryo is Niryo's pythonic interface. The library abstracts ROS1 firmware complexity with clean Python APIs, offers straightforward robot control methods like `move_pose()` and `vision_pick()`, and provides async capabilities through Python's native concurrency primitives. Initial proof-of-concept code demonstrated that single-robot operation and multiple robots can work with python elegantly. 

While the PyNiryo library provides an accessible entry point for controlling the Niryo Ned2, its utility is primarily limited to standalone applications. As project requirements scale from individual units to coordinated multi-robot frameworks, the library’s architectural limitations often create technical deadends that hinder system integration.

See more: [Pyniryo - Documentation](https://niryorobotics.github.io/pyniryo/v1.2.4/examples/examples_basics.html)



## Why PyNiryo is not enough

### The Communication Architecture Gap
PyNiryo operates through TCP connections to the robot's internal server. The `NiryoRobot` class uses `connect()` to establish TCP sessions and sends commands over this proprietary protocol. Examining the API reveals no ROS2 topic publishers, no transform broadcasters, no DDS middleware integration.

Methods like `get_joints()`, `get_pose()`, and `collision_detected` return state synchronously through function calls. There's no mechanism to subscribe to joint states as they update, no way to monitor end-effector poses on a continuous topic, and no broadcast transforms for spatial reasoning.

For multi-robot coordination, this creates problems because

*   **No shared spatial awareness:** Without `tf2` integration, determining "where is the other robot's gripper relative to my workspace?" requires manual coordinate transformations. PyNiryo provides `get_pose()` and `forward_kinematics()`, but these return point-in-time snapshots, not continuous spatial state.
*   **No pub/sub coordination patterns:** Task allocation in multi-robot systems typically uses topic-based queues. One node publishes tasks; multiple robots subscribe and claim them. PyNiryo's TCP model requires either polling both robots constantly or implementing custom message passing, which essentially rebuilds ROS2’s middleware.
*   **Isolated error handling:** When PyNiryo encounters errors, it raises exceptions like `NiryoRobotException`. These are Python-local events. There's no system-level error propagation, no way for one robot to signal the other about collision risks or workspace conflicts through standardised messages.


## What ROS2 Native Actually Provides

The native ROS2 driver shifts the paradigm from request/response control to publish/subscribe middleware. Here is what that architectural difference enables:

### Transform Tree (TF2) as Spatial Database
Both robots publish their complete kinematic chains to `/tf` and `/tf_static` topics. Every joint publishes its transform relative to its parent link. The end-effector's pose propagates through this chain automatically. 

Collision detection becomes a `tf2` query:
```python
# Get transform between robot 1's end effector and robot 2's base
transform = tf_buffer.lookup_transform(
    'robot_2/base_link',
    'robot_1/ee_link',
    rclpy.time.Time()
)
distance = calculate_distance(transform)
```
No manual matrix math. No polling both robots for current poses. The `tf2` system maintains this spatial graph with temporal interpolation.

### Topic-Based State Distribution
Joint states flow on `/robot_1/joint_states` and `/robot_2/joint_states` as `sensor_msgs/JointState` messages. Any node can subscribe. A collision avoidance node can monitor both simultaneously. A task allocator can detect exactly when a robot becomes available based on its status flags. This is fundamentally different from PyNiryo's `get_joints()`, which requires explicit polling.

### Action Servers for Preemptable Operations
ROS2 actions provide goal-oriented, cancelable commands with continuous feedback. A pick-and-place action accepts a target pose, publishes progress updates (approaching, grasping, retracting), and can be preempted if a collision risk emerges. PyNiryo's `pick_from_pose()` is fire-and-forget; once called, the motion completes or crashes.



## The Ecosystem Integration Factor

The native driver unlocks tools designed specifically for ROS2 multi-robot systems:

1.  **MoveIt2:** Motion planning with collision avoidance. It accepts planning scene updates from both robots' joint states and generates trajectories that avoid both self-collision and robot-robot collision.

2.  **Sensor Fusion:** External cameras or other custom sensors can publish to standard message types. Nodes fuse this data with robot state. PyNiryo’s vision system is self-contained and does not integrate with external perception.

3.  **Debugging Tools:** `rqt_graph` visualises node communication; `rviz2` renders the tf tree and robot models in real-time. With PyNiryo, debugging is limited to print statements and custom logging.


## The Tradeoff

**PyNiryo abstracts complexity.** One robot, one script, straightforward control flow. Methods like `pick_and_place()` encapsulate entire sequences. No middleware configuration, no launch files, no QoS policies to understand.

**ROS2 native exposes complexity.** Launch files for namespaced nodes. YAML configs for parameters. Understanding DDS QoS for reliable communication. Manual coordination of multiple launch processes.

But that exposed complexity is **structured complexity.** It involves documented patterns, debuggable with standard tools, and reproducible across projects. Building multi-robot coordination on PyNiryo means creating custom middleware from scratch, covering coordination protocols, state synchronization, and spatial awareness. That is **unstructured complexity** with project-specific solutions.


## The Decision Point: Scaling Beyond the Single Script

The decision to pivot to ROS2 was ultimately a choice between **convenience** and **capability**. 

For our multi-robot Ned2 project, the native ROS2 driver provides a features rich foundation.
