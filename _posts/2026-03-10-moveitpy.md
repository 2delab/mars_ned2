## What MoveItPy Is

MoveItPy is the official Python binding layer for MoveIt2's C++ core via pybind11. Unlike the `move_group` action interface, where a script sends goals to a separately running node over ROS2 actions, MoveItPy embeds the planning stack directly in-process. Planning, scene management, and execution all happen inside the same Python process, with no serialisation overhead between the script and the planner. This is why MARS uses it: the planning scene is a live in-memory object, collision objects written through the Python API are visible to the next `plan()` call immediately, and there is no separately managed node to keep alive.

At startup MoveItPy spawns its own ROS2 node and spins it on a detached background executor thread. ROS2 communications, including joint state subscriptions and controller action clients, are handled on that thread transparently. The planning API is exposed synchronously to the calling thread.

## Initialisation

MARS initialises MoveItPy with a single config file from the `niryo_ned2_dual_arm_moveit_config` package:

```python
config_file = os.path.join(
    get_package_share_directory("niryo_ned2_dual_arm_moveit_config"),
    "config",
    "moveit_py_params.yaml",
)
moveit_py = MoveItPy(node_name="moveit_py_test", launch_params_filepaths=[config_file])
```

At startup, MoveItPy reads the URDF and SRDF from ROS2 parameters, builds the robot model, and blocks until the first joint state message arrives on `/joint_states`. That topic is published by `JointStateManager`, which aggregates hardware state for all 12 joints of both arms under prefixed names. If `JointStateManager` is not running when MoveItPy starts, the planner has no state to plan from. The config sets a 15-second timeout before the node continues with an undefined initial state:

```yaml
planning_scene_monitor_options:
  robot_description: "robot_description"
  robot_description_semantic: "robot_description_semantic"
  joint_state_topic: "/joint_states"
  wait_for_initial_state_timeout: 15.0
```

### Why MARS uses three instances

`MoveItPy.execute()` is blocking; it holds the calling thread until the trajectory completes or fails. A single instance executing one arm's trajectory cannot simultaneously process the other arm's execution on the same background executor. To run both arms concurrently, MARS creates three independent instances:

- `moveit_py_arm1`: scoped to the single-arm config, owns `arm_1`'s planning component and action client
- `moveit_py_arm2`: same for `arm_2`
- `moveit_py_dual`: scoped to the dual-arm config, used for joint-space goals on the `dual` group

Each instance has its own background executor thread, its own controller action clients, and its own `PlanningSceneMonitor`. This last point has a direct operational consequence: the workspace collision object must be applied to each instance's scene independently. An object written into `moveit_py_dual`'s scene does not appear in `moveit_py_arm1`'s scene.

## Configuration

### Planning Pipelines and the Adapter Chain

Both OMPL and Pilz Industrial Motion Planner are loaded at startup:

```yaml
planning_pipelines:
  pipeline_names: ["ompl", "pilz_industrial_motion_planner"]
  default_planning_pipeline: "ompl"
```

What OMPL returns from a planning call is a raw path: a sequence of joint-space waypoints that are collision-free but carry no timing information (no timestamps, no velocities, no accelerations). That output cannot be sent to a hardware controller directly. The response adapter chain is what converts it into an executable trajectory:

```yaml
ompl:
  response_adapters:
    - default_planning_response_adapters/AddTimeOptimalParameterization
    - default_planning_response_adapters/ValidateSolution
    - default_planning_response_adapters/DisplayMotionPath
```

`AddTimeOptimalParameterization` takes the raw waypoint sequence and fits time-optimal velocity and acceleration profiles to it, using the per-joint limits declared in the config as hard ceilings. The path shape (which joint-space configurations to pass through) is fixed by OMPL. The timing (how fast to move through each segment) is computed entirely by this adapter. The request adapters run before planning:

```yaml
  request_adapters:
    - default_planning_request_adapters/ResolveConstraintFrames
    - default_planning_request_adapters/ValidateWorkspaceBounds
    - default_planning_request_adapters/CheckStartStateBounds
    - default_planning_request_adapters/CheckStartStateCollision
```

`CheckStartStateCollision` rejects planning attempts where the current joint state is already in collision, which would happen if the robot had been moved manually or if the planning scene had been updated with a new obstacle that now overlaps the robot's current pose.

Named parameter sets (`pilz_lin`, `pilz_ptp`) are pre-defined profiles that switch to the Pilz pipeline at the call site. Pilz LIN produces Cartesian-linear paths, meaning straight-line tool motion in task space, which OMPL cannot guarantee. These are available for single-arm planning but not for the `dual` group, for reasons explained below.

### Joint Limits and Velocity Scaling

`AddTimeOptimalParameterization` uses the per-joint velocity and acceleration limits as its hard ceilings:

```yaml
robot_description_planning:
  joint_limits:
    arm_1_joint_1:
      has_velocity_limits: true
      max_velocity: 1.57        # rad/s
      has_acceleration_limits: true
      max_acceleration: 2.0     # rad/s²
    arm_1_joint_2:
      max_velocity: 1.047       # mechanically limited shoulder joint
      max_acceleration: 0.8
    arm_1_joint_4:
      max_velocity: 3.14        # wrist joints, less inertia
      max_acceleration: 3.0
    # mirrored for arm_2
```

The `max_velocity_scaling_factor: 0.7` in `plan_request_params` multiplies these limits before the adapter runs. The effective ceiling for `arm_1_joint_2` during a default plan call is `1.047 × 0.7 = 0.733 rad/s`. Individual `plan()` calls can pass a lower scaling factor to reduce this further; they cannot exceed the config default without changing the config. The asymmetry across joints reflects the physical robot: the shoulder joint (`joint_2`) carries the full arm mass and is mechanically limited; the wrist joints (`joint_4`–`joint_6`) have less inertia and can move faster.

### Kinematics

KDL is configured independently for each arm group:

```yaml
robot_description_kinematics:
  arm_1:
    kinematics_solver: kdl_kinematics_plugin/KDLKinematicsPlugin
    kinematics_solver_search_resolution: 0.005
    kinematics_solver_timeout: 0.005
  arm_2:
    kinematics_solver: kdl_kinematics_plugin/KDLKinematicsPlugin
    kinematics_solver_search_resolution: 0.005
    kinematics_solver_timeout: 0.005
```

KDL uses an iterative Jacobian pseudoinverse method on each arm's own 6-DOF kinematic chain. There is no IK solver configured for the `dual` group. This is not an oversight; the `dual` group plans in a 12-DOF joint space where the goal is expressed as a `RobotState` with joint positions for both arms, not as a Cartesian pose. IK is only needed when converting a Cartesian goal to joint angles, and that conversion happens at the single-arm level.

## The Workspace Collision Object

Before any planning, MARS adds a 3×3 m floor box to each instance's planning scene to represent the physical workspace platform. Without it the planner has no knowledge of the surface and may return trajectories that pass through it.

```python
platform = CollisionObject()
platform.header.frame_id = "world"
platform.id = "workspace_platform"
platform.operation = bytes([0])  # ADD

primitive = SolidPrimitive()
primitive.type = SolidPrimitive.BOX
primitive.dimensions = [3.0, 3.0, 0.01]
platform.primitives.append(primitive)

pose = Pose()
pose.position.z = -0.015
pose.orientation.w = 1.0
platform.primitive_poses.append(pose)

with moveit_py.get_planning_scene_monitor().read_write() as scene:
    scene.apply_collision_object(platform)
```

`read_write()` acquires a write lock on the in-process planning scene for the duration of the block. The change is visible to all subsequent `plan()` calls on that instance immediately. Because each `MoveItPy` instance maintains its own scene, this block must be executed once for each of the three instances (`moveit_py_arm1`, `moveit_py_arm2`, and `moveit_py_dual`) before any planning begins.

## Planning and Execution

### Joint-space goals with the dual group

For moves where both arms' targets are expressed in joint space, MARS plans against the `dual` group. A `RobotState` carries the target positions for both arms and is passed as the goal:

```python
dual_arm = moveit_py_dual.get_planning_component("dual")
robot_model = moveit_py_dual.get_robot_model()

robot_state = RobotState(robot_model)
robot_state.set_joint_group_positions("arm_1", [0, 0, 0, 0, 0, 0])
robot_state.set_joint_group_positions("arm_2", [0, 0, 0, 0, 0, 0])

dual_arm.set_start_state_to_current_state()
dual_arm.set_goal_state(robot_state=robot_state)

plan_result = dual_arm.plan()
if plan_result:
    moveit_py_dual.execute(plan_result.trajectory, controllers=[])
```

`set_start_state_to_current_state()` reads the current joint positions from the planning scene monitor, fed by `/joint_states`, and sets them as the planning start. This must be called before each new plan. After execution the robot is at a new configuration; skipping this call causes the planner to treat the previous goal as the start, which diverges from the actual post-execution state and produces plans that begin with a jump.

`execute(plan_result.trajectory, controllers=[])` passes the trajectory to `TrajectoryExecutionManager`. `controllers=[]` instructs it to select the controller from the config by matching the trajectory's joint list against registered controller entries; the 12-joint dual trajectory matches the `dual` controller entry and dispatches to both arm action servers simultaneously.

### Cartesian goals: why the dual group cannot be used

`set_goal_state(pose_stamped_msg=..., pose_link=...)` internally calls the IK solver configured for the planning group to convert the Cartesian pose to a joint configuration. The `dual` group has no IK solver configured; KDL is only configured for `arm_1` and `arm_2`. The call fails at the solver lookup. This is not an API-level restriction that could be worked around; it reflects the fact that a single IK solve cannot simultaneously satisfy two independent end-effector poses in a 12-DOF system without additional constraints.

There are two approaches to dual-arm Cartesian goals:

**Option A: Solve IK externally, use the dual group for planning.** Call `set_joint_group_positions` on a `RobotState` with IK solutions for each arm obtained separately via `arm_1`/`arm_2` planning components, then pass that `RobotState` as the goal to the `dual` group. This preserves the 12-DOF shared planning with inter-arm collision checking and produces a single trajectory with identical timestamps for both arms. The cost is implementation complexity: IK failure for either arm requires replanning from scratch, and IK solutions must be checked for mutual reachability before committing to the dual plan.

**Option B: Plan each arm independently.** What MARS does. Each arm's planning component takes a `PoseStamped` goal directly, plans in its own 6-DOF space, and executes on its own `MoveItPy` instance concurrently via threads:

```python
arm_1.set_start_state_to_current_state()
arm_2.set_start_state_to_current_state()
arm_1.set_goal_state(pose_stamped_msg=ps1, pose_link="arm_1_tool_link")
arm_2.set_goal_state(pose_stamped_msg=ps2, pose_link="arm_2_tool_link")

plan_result_1 = arm_1.plan()
plan_result_2 = arm_2.plan()

if plan_result_1 and plan_result_2:
    t1 = threading.Thread(target=lambda r=plan_result_1: moveit_py_1.execute(r.trajectory, controllers=[]))
    t2 = threading.Thread(target=lambda r=plan_result_2: moveit_py_2.execute(r.trajectory, controllers=[]))
    t1.start(); t2.start()
    t1.join(); t2.join()
```

This also unlocks Pilz LIN for each arm; passing `plan_request_params=pilz_lin` to `arm_1.plan()` produces a Cartesian-linear path for that arm, something OMPL cannot guarantee regardless of group. The trade-off is that the two trajectories have independent time parameterisations: each arm's duration is determined by its own path length and joint limits. The arms start simultaneously but are not guaranteed to arrive at their goals at the same instant. Their relative motion through the workspace during execution is not jointly planned, which means inter-arm collision safety during execution relies on the static snapshot taken at planning time; each arm's plan was checked against the other arm's position as it was when planning ran, not as it will be at each moment during concurrent execution.

## What MoveItPy Does Not Provide

Once `execute()` is called, the trajectory runs as open-loop joint interpolation on the hardware controller. The planning scene continues updating from `/joint_states` during execution, but nothing in MoveItPy reads it for execution safety. An obstacle added after the planning call completes will not halt execution. A joint state drift that puts the robot on a collision course mid-trajectory will not be detected. The planning guarantee, that the trajectory was collision-free in the planning scene at planning time, is the only guarantee MoveItPy provides.
