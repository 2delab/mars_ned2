## Simulation for reinforcement learning

MuJoCo (Multi-Joint dynamics with Contact) is a physics engine optimised for control, reinforcement learning, and dynamics research. 

**Core design:**
- Fast implicit contact dynamics (designed for contact-rich tasks: grasping, pushing, assembly)
- Fixed timestep, no adaptive stepping (predictable, deterministic, JAX-compilable)
- Native Python bindings and a C API
- No scenegraph, no rendering pipeline, no sensor simulation framework

You describe your robot as an MJCF file (MuJoCo model format). 

## URDF → MJCF: What Gets Lost in Translation

Used MuJoCo's built-in URDF loader:

```python
import mujoco

# Load URDF (must be in same directory as meshes or use meshdir)
model = mujoco.MjModel.from_xml_path('niryo_ned2.urdf')

# Save as MJCF
mujoco.mj_saveLastXML('niryo_ned2_converted.xml', model)
```

**Key considerations**:
- MuJoCo automatically converts URDF elements to MJCF equivalents
- Inertial properties are validated and auto-corrected if needed
- Joint axes and limits are preserved
- Quaternion orientations are computed from RPY values

![MUJOCO model](/mars_ned2/assets/images/mujoco.png){: .align-center}

This conversion is **not lossless**. 

some examples of what changes include :


### 1. Actuator Dynamics

URDF specifies `<effort>` and `<velocity>` limits per joint. MJCF requires explicit actuator models: position control, velocity control, or torque control, with stiffness, damping, and control range.

```
<!-- URDF: abstract limits -->
<limit effort="50" velocity="1.0"/>

<!-- MJCF: explicit control model -->
<actuator>
  <position name="joint_1" joint="joint_1" kp="100" kv="10" ctrlrange="-2.999 2.999"/>
</actuator>
```

The `kp` (position gain) and `kv` (damping) are not measured on the hardware — they are estimated. A policy trained with `kp=100, kv=10` is learning how to control an idealized position-controlled joint. The NED2's actual hardware has different feedback control, different acceleration limits, and different tracking error.

### 2. Friction and Contact

URDF meshes specify a visual geometry. MJCF requires separate collision geometry  and surface properties: friction, elasticity, damping.

The NED2's gripper has rubber-tipped fingers. Rubber on steel has friction coefficient μ ≈ 0.8–1.2. I estimated 0.8. I never measured it. The conversion tool cannot know this; it is human knowledge.

A grasp that works with friction 0.8 may fail with friction 0.7 and succeed with friction 0.9. An RL policy trained with friction 0.8 encodes that assumption in its learned behaviour.



## Commanding a Joint: MuJoCo vs ROS2

### In MuJoCo

```python
model = mujoco.MjModel.from_xml_path("niryo.mjcf")
data = mujoco.MjData(model)

# Set desired position for joint_1
target_pos = 0.5  # radians
data.ctrl[0] = target_pos

# Step the physics
mujoco.mj_step(model, data)

# Check actual position
actual_pos = data.qpos[0]
print(f"Target: {target_pos}, Actual: {actual_pos}")
```

You are directly commanding a physics simulator's actuator. The command is processed by the position controller (with gains `kp`, `kv`) and the resulting torque is applied in the next timestep. You have direct, low-level control of the simulation loop.

### In ROS2 with Classical Planning

```python
# (Pseudocode; actual code is more verbose)
move_group = MoveGroupInterface(arm_name="arm_1")

# Planning: where should the arm go?
goal_pose = Pose(position=[0.3, 0.0, 0.2], orientation=[...])
plan = move_group.compute_plan(goal_pose)
# plan contains waypoints with timing

# Execution: send the plan to hardware
trajectory = plan.trajectory
action_client.send_goal(trajectory)  # FollowJointTrajectory action
action_client.wait_for_result()
```

You are commanding a hardware controller with a *trajectory*, not a setpoint. The hardware driver interpolates the waypoints, generates its own velocity profile, and executes it asynchronously. You are one level of abstraction higher: planning trajectories, not setting joint positions.


### The Difference

**MuJoCo:** Low-level control loop. You set actuator commands every 2 ms and read state back. This is what a real hardware control loop does, but at zero latency and with perfect state information.

**ROS2:** High-level planning. You compute a safe trajectory offline, send it to hardware, and the hardware executes it independently. The trajectory absorbs parameter uncertainty through conservative design (collision padding).


## The Sim-to-Real Gap

Even if we had a perfect MuJoCo model wih perfect inertia, perfect friction and perfect contact dynamics, there is a gap between the simulation and the Real deployment.

### MuJoCo Assumptions

- Commands are issued at 500 Hz (2 ms timestep)
- Commands execute instantaneously or within the next timestep
- You observe the outcome of your command in the next timestep
- Closed-loop feedback is available at 500 Hz

### NED2 Hardware Reality

- Commands are issued over TCP/IP (20–50 ms round-trip)
- The hardware driver publishes joint state at 15 Hz (67 ms updates)
- By the time you see the outcome, 50+ ms have passed
- Closed-loop feedback is available at 15 Hz

### Why this matters

An RL policy (trained with PPO, SAC, or any other algorithm) learns the relationship between actions and observations. when trained in simulation, it learns:

```
action_t → [instant physics integration] → observation_{t+1} at t+2ms
```

On hardware, the actual relationship is:

```
action_t → [50ms network latency] → [67ms sensor publication] → observation_{t+1} at t+117ms
```

The policy has learned to predict 2 ms into the future and adapt. It has **not** learned to predict 117 ms into the future. It has not learned to wait for delayed feedback before issuing the next command.

This is not a dynamics problem or a friction problem. It is a temporal problem. The policy learned the wrong temporal model of the system.

Domain randomisation can help — train with randomised latencies, command frequencies, and sensor delays. But this requires:
1. Knowing the actual distribution of latencies on hardware (measurement)
2. Training with enough variation that the policy generalises (millions of steps, not 100,000)
3. Validating on real hardware that the randomisation was adequate

Without these, the MuJoCo policy is fundamentally mismatched to the hardware.


**NED2:**  has a built-in controller, but you interact with it via ROS2 drivers over TCP/IP. The control latency is determined by network round-trip time, not by the robot's internal control loop. The robot publishes joint state asynchronously, not on a hard real-time schedule.

The NED2 was never designed for microsecond-precision control. It was designed for cobot safety (force-limited, slow-moving). The control architecture trades precision for safety.



## Why MARS Uses Kinematic Planning Instead

Classical motion planning (MoveIt2, geometric collision checking) avoids the sim-to-real gap by **not relying on simulated dynamics**.

Here is what we assume:
1. The robot can execute a commanded trajectory to within its repeatability specification (±0.5 mm for NED2)
2. Joint-space trajectories are sufficient; we do not need to model contact dynamics or friction

which means

- Perfect physics fidelity is not needed. Kinematic planning uses geometric models, not contact models.
- Latency does not break safety. 
- The approach works for any robot that has a position controller. You do not need to know its internal control loop.

An RL policy, by contrast, is **sensitive** to every parameter: friction, latency, actuator dynamics, contact stiffness.


## The Tradeoff

**MuJoCo (and RL in general):**
- Pros: Can learn complex dexterous skills, adapt to dynamic environments, potentially faster execution
- Cons: Requires high-fidelity simulation, domain randomisation, millions of training steps, real hardware validation

**Kinematic Planning:**
- Pros: Works with approximate models, tolerates latency, scales to multi-arm systems, proven in industry
- Cons: Cannot solve manipulation tasks that require fine contact control (dexterous grasping, tool manipulation)

MARS solves the **coordination** problem by keeping two arms aware of each other and preventing collisions. It assumes arm-level controllers are available and working. It does not solve the **manipulation** problem. 

An RL approach would solve the manipulation problem but create the coordination problem. You would need to learn a policy for two arms simultaneously, which is even higher-dimensional and more sample-inefficient than a single arm.

By choosing planning over learning, MARS would enable robots be coordinated safely and efficiently through a conservative geometric planning and real-time state aggregation.
