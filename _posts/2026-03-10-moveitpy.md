# MoveIt's python API

MoveItPy is the official Python binding layer for MoveIt2's C++ core, implemented via pybind11. It is not a wrapper around the `move_group` action interface. Where the action interface communicates with a separately running `move_group` node over ROS2 actions, MoveItPy embeds the MoveIt2 planning stack directly in-process — planning, scene management, and execution all happen inside the same Python process.

The consequence is that there is no inter-process serialisation between your script and the planner. The planning scene is a live in-memory object you can read and write directly. Collision objects added through the Python API are visible to the next `plan()` call without any ROS2 publish/subscribe round-trip.

When `MoveItPy` is instantiated it starts its own ROS2 node, spins that node on a detached `SingleThreadedExecutor` background thread, and then exposes the planning API synchronously to the calling thread. From that point, planning and execution are driven by the calling code — the background thread handles ROS2 communications (joint state updates, controller action clients) transparently.

## Configuration: `moveit_py_params.yaml`

Everything MoveItPy needs is declared in a single params file passed at construction time. The file is loaded from the `niryo_ned2_dual_arm_moveit_config` package:

```python
config_file = os.path.join(
    get_package_share_directory("niryo_ned2_dual_arm_moveit_config"),
    "config",
    "moveit_py_params.yaml",
)
moveit_py = MoveItPy(node_name="moveit_py_test", launch_params_filepaths=[config_file])
```

The file is structured into several independent sections.

### Planning Scene Monitor

```yaml
planning_scene_monitor_options:
  robot_description: "robot_description"
  robot_description_semantic: "robot_description_semantic"
  joint_state_topic: "/joint_states"
  wait_for_initial_state_timeout: 15.0
```

`robot_description` and `robot_description_semantic` are the ROS2 parameter names under which the URDF and SRDF are published by `robot_state_publisher`. MoveItPy reads these at startup to build the robot model.

`joint_state_topic: /joint_states` is the aggregated topic produced by `JointStateManager` — the single topic carrying prefixed joint names for all 12 active joints of both arms. The planning scene monitor subscribes here to keep its internal robot state current. This is why the `JointStateManager` must be running before MoveItPy is used — if no message has arrived on this topic, the monitor has no state to plan from.

`wait_for_initial_state_timeout: 15.0` controls how long MoveItPy blocks at startup waiting for that first joint state message. If the timeout expires without a message the node continues but the initial state is undefined.

### Planning Pipelines

```yaml
planning_pipelines:
  pipeline_names: ["ompl", "pilz_industrial_motion_planner"]
  default_planning_pipeline: "ompl"

plan_request_params:
  planning_pipeline: ompl
  planner_id: "RRTConnectkConfigDefault"
  max_velocity_scaling_factor: 0.7
  max_acceleration_scaling_factor: 0.7
  planning_time: 1.0
```

Both pipelines are loaded simultaneously at startup. The `default_planning_pipeline` determines which is used when `plan()` is called with no explicit parameters. Named parameter sets (`pilz_lin`, `pilz_ptp`) are pre-built profiles that can be passed directly to `plan()` to switch pipeline at the call site.

The OMPL section configures request and response adapter chains. The key adapter on the response side is `AddTimeOptimalParameterization` — OMPL finds a collision-free path shape in joint space, and this adapter fits time-optimal velocity/acceleration profiles to the waypoints, respecting the joint limits declared elsewhere in the file. The path shape and the timing are computed separately.

`longest_valid_segment_fraction: 0.005` means the planner performs collision checks at every 0.5% increment of normalised path length — approximately every 0.03 rad per joint at typical travel distances. This is what makes planning-time collision avoidance a hard geometric guarantee: a trajectory only reaches execution if every point along it is collision-free.

### Kinematics

```yaml
robot_description_kinematics:
  arm_1:
    kinematics_solver: kdl_kinematics_plugin/KDLKinematicsPlugin
    kinematics_solver_search_resolution: 0.005
    kinematics_solver_timeout: 0.005
  arm_2:
    kinematics_solver: kdl_kinematics_plugin/KDLKinematicsPlugin
    kinematics_solver_search_resolution: 0.005
    kinematics_solver_timeout: 0.005
```

KDL (Kinematics and Dynamics Library) is configured independently for each arm group. It operates on each arm's own 6-DOF kinematic chain using an iterative Jacobian pseudoinverse method. Cross-arm collision awareness is not part of IK — that happens at the OMPL planning level where the full planning scene including both arms is visible.

`kinematics_solver_search_resolution: 0.005` rad is the step size for the iterative search. `kinematics_solver_timeout: 0.005` s is the per-call timeout. For Cartesian goals, IK is called to convert the Cartesian pose to a joint configuration before planning begins in joint space.

### Joint Limits

```yaml
robot_description_planning:
  joint_limits:
    arm_1_joint_1:
      has_velocity_limits: true
      max_velocity: 1.57
      has_acceleration_limits: true
      max_acceleration: 2.0
    arm_1_joint_2:
      max_velocity: 1.047   # shoulder, mechanically limited
      max_acceleration: 0.8
    arm_1_joint_4:
      max_velocity: 3.14    # wrist joints
      max_acceleration: 3.0
    # ... mirrored for arm_2
```

Per-joint velocity and acceleration limits are declared for all 12 active joints. The `AddTimeOptimalParameterization` adapter uses these as hard ceilings when fitting trajectory timing. The `max_velocity_scaling_factor: 0.7` in the plan request params scales these limits down at runtime — a system-wide safety margin that individual requests can further reduce.

### Controller Manager

```yaml
moveit_controller_manager: moveit_simple_controller_manager/MoveItSimpleControllerManager

moveit_simple_controller_manager:
  controller_names:
    - arm_1
    - arm_2
    - dual
  arm_1:
    type: FollowJointTrajectory
    joints: [arm_1_joint_1, ..., arm_1_joint_6]
    action_ns: follow_joint_trajectory_prefixed
  arm_2:
    type: FollowJointTrajectory
    joints: [arm_2_joint_1, ..., arm_2_joint_6]
    action_ns: follow_joint_trajectory_prefixed
  dual:
    type: FollowJointTrajectory
    joints: [arm_1_joint_1, ..., arm_1_joint_6, arm_2_joint_1, ..., arm_2_joint_6]
    action_ns: follow_joint_trajectory_prefixed
```

Three controller entries are registered. `TrajectoryExecutionManager` selects the controller that owns the joints in a planned trajectory. A trajectory planned against the `dual` group contains all 12 joints — it matches the `dual` controller entry, which dispatches to `follow_joint_trajectory_prefixed` under both arm namespaces. A trajectory planned against `arm_1` alone matches only the `arm_1` controller entry.

`action_ns: follow_joint_trajectory_prefixed` is the action server name within each arm's namespace — so MoveIt2 sends goals to `/arm_1/follow_joint_trajectory_prefixed` and `/arm_2/follow_joint_trajectory_prefixed`.

## The SRDF: `niryo_ned2_dual_arm.srdf`

The SRDF encodes planning semantics that the URDF cannot express. It has three responsibilities in this system.

### Planning Groups

```xml
<group name="arm_1">
    <joint name="arm_1_joint_world"/>
    <joint name="arm_1_joint_1"/>
    ...
    <joint name="arm_1_joint_6"/>
</group>
<group name="arm_2">
    <joint name="arm_2_joint_world"/>
    ...
</group>
<group name="dual">
    <group name="arm_1"/>
    <group name="arm_2"/>
</group>
```

`arm_1` and `arm_2` are independent 6-DOF kinematic chains. The `dual` group is declared by subgroup inclusion — it does not enumerate joints directly, it composes the two single-arm groups. This gives the `dual` group a 12-DOF joint space. The planner sees both arms as a single system when planning against `dual`, which is what enables inter-arm collision checking during planning.

### Link Padding

```xml
<link_padding link="arm_1_base_link"     padding="0.05"/>
<link_padding link="arm_1_arm_link"      padding="0.05"/>
<link_padding link="arm_1_forearm_link"  padding="0.05"/>
<link_padding link="arm_1_wrist_link"    padding="0.05"/>
<!-- end-effector and sensor links -->
<link_padding link="arm_1_base_gripper_1" padding="0.07"/>
<link_padding link="arm_1_mors_1"         padding="0.07"/>
<link_padding link="arm_1_camera_link"    padding="0.07"/>
<!-- mirrored for arm_2 -->
```

Padding inflates each link's collision geometry by the specified amount. FCL does not check the physical mesh surface — it checks the padded volume. Standard arm links receive 0.05 m; end-effector and sensor links receive 0.07 m to account for their larger effective geometry.

Because padding is symmetric, two opposing standard links trigger a collision detection event when their physical surfaces are separated by as little as 10 cm (0.05 + 0.05 m). End-effector pairs trigger at up to 14 cm. The planner routes every trajectory through configurations that satisfy this clearance — it is a geometric constraint enforced during planning, not a runtime check.

### Collision Exclusions

```xml
<!-- Adjacent: kinematically connected, physical overlap is by design -->
<disable_collisions link1="arm_1_arm_link"      link2="arm_1_elbow_link"    reason="Adjacent"/>
<disable_collisions link1="arm_1_forearm_link"  link2="arm_1_wrist_link"    reason="Adjacent"/>

<!-- Never: geometrically impossible to collide given the robot's kinematics -->
<disable_collisions link1="arm_1_base_link"     link2="arm_2_forearm_link"  reason="Never"/>
<disable_collisions link1="arm_1_arm_link"      link2="arm_2_base_link"     reason="Never"/>
```

145 pairs are disabled in total. Without these, every planning sample would trigger an O(n²) narrow-phase test across all link combinations in the system — for 22 links per arm pair, that is potentially hundreds of GJK tests per configuration sample. Disabling the 145 known-safe pairs reduces narrow-phase tests per sample by approximately 75%. In a 12-DOF search space where RRTConnect generates thousands of samples per planning call, this reduction is significant.

The `Adjacent` category covers kinematically connected links where overlap is structurally intentional. The `Never` category covers pairs that are geometrically separated by enough kinematic distance that collision is impossible regardless of joint configuration.

## Collision Objects in Code

The SRDF encodes the robot's own geometry and the relationships between its links. It has no mechanism for expressing the external environment. Surfaces, tables, workspace boundaries — anything that exists in the physical world but is not part of the robot description — must be added to the planning scene programmatically as `CollisionObject` messages.

### Why add them in code rather than at launch

MoveItPy's `PlanningSceneMonitor` exposes direct read-write access to the live in-process planning scene. Adding collision objects through the Python API writes them into the same scene object that the planner queries. No ROS2 publish/subscribe round-trip is involved. The change is visible to the planner on the next `plan()` call immediately.

This is different from the `move_group` action interface, where collision objects must be published to the planning scene topic and the external `move_group` node updates its own scene copy asynchronously. With MoveItPy, the scene and the planner share memory.

### `workspace_scene.yaml`

```yaml
collision_objects:
  - id: "workspace_platform"
    header:
      frame_id: "world"
    primitives:
      - type: 1        # BOX
        dimensions: [3.0, 3.0, 0.01]
    primitive_poses:
      - position: {x: 0.0, y: 0.0, z: -0.015}
        orientation: {x: 0.0, y: 0.0, z: 0.0, w: 1.0}
    operation: 0       # ADD
```

The workspace floor is a 3m × 3m × 10mm box centred at z = -0.015 m in the world frame — just below the robot bases. `type: 1` is the `SolidPrimitive` constant for BOX. `operation: 0` is ADD. The geometry is a primitive rather than a mesh because primitive collision shapes are cheaper to check against and sufficient for a flat workspace surface.

### The `read_write()` context manager

```python
planning_scene_monitor = moveit_py.get_planning_scene_monitor()

with planning_scene_monitor.read_write() as scene:
    for obj_data in scene_data["collision_objects"]:
        collision_obj = create_collision_object(obj_data)
        scene.apply_collision_object(collision_obj)
```

`read_write()` acquires a write lock on the in-process planning scene for the duration of the `with` block. `apply_collision_object()` inserts the object. The lock is released on exit. All subsequent `plan()` calls — for either arm group, synchronous or asynchronous — will treat the workspace floor as a hard collision obstacle. Trajectories that would intersect it are rejected at planning time.

Without this object, the planner has no knowledge of the physical platform. The robot's links would be checked against each other and against the SRDF-declared environment, but nothing prevents the planner from generating a trajectory that passes through the floor surface.

## The Execution Flow: `moveit5.py`

The implementation demonstrates synchronous dual-arm execution using the `dual` planning group.

**Initialisation**

```python
moveit_py = MoveItPy(node_name="moveit_py_test", launch_params_filepaths=[config_file])
load_and_add_collision_objects(moveit_py)

dual_arm = moveit_py.get_planning_component("dual")
robot_model = moveit_py.get_robot_model()
```

`MoveItPy` starts the background executor and waits for the initial joint state. `load_and_add_collision_objects` adds the workspace floor before any planning occurs. `get_planning_component("dual")` returns a `PlanningComponent` scoped to the 12-DOF dual group — this is the object that holds planning context (start state, goal state) between calls.

**Setting a goal state**

```python
robot_state = RobotState(robot_model)
robot_state.set_joint_group_positions("arm_1", [0, 0, 0, 0, 0, 0])
robot_state.set_joint_group_positions("arm_2", [0, 0, 0, 0, 0, 0])
dual_arm.set_goal_state(robot_state=robot_state)
```

`RobotState` is a complete configuration of the robot — all joints. `set_joint_group_positions` populates the positions for the named group's joints within that state. Both arms' goal positions are set on the same `RobotState` object, which is then passed as the goal for the `dual` planning group. The planner searches for a path in 12-DOF space from the current state to this combined goal.

**Planning and executing**

```python
dual_arm.set_start_state_to_current_state()
plan_result = dual_arm.plan()

if plan_result:
    moveit_py.execute(plan_result.trajectory, controllers=[])
```

`set_start_state_to_current_state()` reads the current joint positions from the planning scene monitor (fed by `/joint_states`) and sets them as the planning start. `plan()` runs OMPL RRTConnect in the 12-DOF space, applies the adapter chain, and returns a `MotionPlanResponse`. 

`execute(plan_result.trajectory, controllers=[])` passes the trajectory to `TrajectoryExecutionManager`. `controllers=[]` instructs it to use the controller manager configuration to determine which controller to use — it matches the trajectory's joint list against the registered controller entries and selects `dual`, which dispatches to both arm action servers. The Python GIL is released during execution, and `execute()` returns when the trajectory completes or fails.

**Chaining moves**

```python
# Phase 2: set new goal, plan from current state, execute
dual_arm.set_start_state_to_current_state()
robot_state = RobotState(robot_model)
target_radians = math.radians(50.0)
robot_state.set_joint_group_positions("arm_1", [target_radians, 0, 0, 0, 0, 0])
robot_state.set_joint_group_positions("arm_2", [target_radians, 0, 0, 0, 0, 0])
dual_arm.set_goal_state(robot_state=robot_state)
plan_result = dual_arm.plan()
if plan_result:
    moveit_py.execute(plan_result.trajectory, controllers=[])
```

`set_start_state_to_current_state()` is called again before each new plan. After execution the robot is at a new configuration — the planning scene monitor has updated via `/joint_states`. Calling this method ensures the next plan starts from the actual post-execution state rather than the previous goal. Skipping this would cause the planner to plan from where it thought the robot would be, which may diverge from where it actually is.
